Component 01 — Keystone

Contract Parser

PRD Complete
Methodology Root

Maps the BBX 4-Phase Planning Tool directly. Phase 01 Prepare: Objectives, BATNA, Alignment/Conflict. Terms Planning: Get/Give/Both, Priority, Walkaway/Goal zones. Phase 02 Communicate: Information strategy, Domain Glossary. Phase 03 Propose: Trading strategy, Packages. Phase 04 Align: Position tracker, Problem planner.

What Ali Generates

Pre-populated negotiation workbook from uploaded contract. Domain glossary with definitions + negotiation significance. Approach classification (which of 5 types). Risk scoring per clause. Pattern Mirror: "Here's what Ali sees. Here's what Ali doesn't see. Fill the gaps."

Existing Data

Anonymize script (scripts/anonymize-contract.py) — Stage 1 done. 4 real anonymized client files ready. Full PRD with extraction schema, 3-stage pipeline architecture, and competitive analysis. Parsing is commodity (Docling/LlamaParse). The schema — the methodology as machine-executable JSON — is the product.

Stakeholder Input Needed

10 open questions in the PRD. Most critical: Validate the JSON extraction schema (does it match how contracts are decomposed?). Annotate 1 anonymized contract as ground truth. Confirm Terms vs. Clauses decomposition model.

Component 02

Power Dynamics

Types Defined
Methodology Root

Core pillar of ASF: Three Forces of Leverage. Market forces (Porter's Five, loss aversion). External forces (counterparty BATNA, stakeholder map, constraints). Internal forces (team alignment, role clarity, unified messaging). Connected to the Inversion Method: map how the deal dies, then avoid those paths.

What Ali Generates

3-panel force assessment with scores. Counterparty power map: "CEO has final say, CFO has veto, procurement recommends." Leverage gap: "Weak on price (market saturated) but strong on timeline (they need Q2 close)."

Existing Data

ForcesAssessment TypeScript interface defined with market/external/internal fields. Classification signals capture alternativesSaturation, timeConstraint. Full ASF Force sections chunked and injectable via knowledge layer.

Stakeholder Input Needed

The 5 axes in the viz (Information, Alternatives, Time Pressure, Relationship, Authority) — are these the right dimensions? Or should we use Market/External/Internal from ASF directly? How are forces scored in practice?

Component 03

ZOPA Analysis

Framework Ready
Methodology Root

Tom's Goal Zone Framework: Goal Zone (optimal) → Authorization Zone (acceptable with justification) → Why Zone (floor, requires explanation) → Walkaway (absolute boundary). Built on Fisher & Ury BATNA theory. Applies across all 4 negotiation types equally.

What Ali Generates

Your ZOPA band with 4 zones mapped. Counterparty estimated range (inferred or provided). Overlap detection: "$100K-$150K — the sweet spot." Risk alert: "If walkaway is $85K and they won't go below $95K — zero ZOPA. Improve your BATNA or change scope."

Existing Data

Goal Zone framework fully encoded in coaching methodology. User confidence fields (offers, preparation, closing) in schema. extractedEntities.dealValue captures amounts. No structured ZOPA model yet — zones exist in methodology but not as captured user data.

Stakeholder Input Needed

Is the Zones / RPG framework proprietary? Can Ali reference Goal/Authorization/Why/Walkaway by name? How are these zones calculated for clients — is there a formula or is it expert judgment? This is a pre-beta blocker.

Component 04

Offer Flow Timeline

Needs Schema
Methodology Root

ASF Phase 3: PROPOSE. Built on the Anchor Drop move (first offer sets reference frame), Conditional Trading ("If you X, then we Y"), and the ever-decreasing concessions principle. Forces of Leverage inform acceptable ranges.

What Ali Generates

Turn-by-turn offer tracking. Concession rate analysis: "They dropped $5K. Your move should be smaller to signal firmness." Convergence prediction: "At this rate, you'll hit walkaway in 3 rounds."

Existing Data

ChatMessage stores raw text. Conversation.extractedEntities captures dealValue (amount, currency, range). No structured offer history — each message is atomic text, no versioned offer chain.

Stakeholder Input Needed

How are concession patterns tracked today? What triggers a "too aggressive" vs. "too soft" concession? Per-term or aggregate offers? We need the data model for the offer chain.

Component 05

Skill Development Tracker

Snapshot Only
Methodology Root

Tom's 4-Phase framework maps directly to competencies: PREPARE → Research & goal-setting. COMMUNICATE → Rapport & discovery questions. PROPOSE → Anchoring & objection handling. ALIGN → Securing commitment. Each Aligned Move builds a specific capability.

What Ali Generates

Skill gap analysis: "Strong in Preparation (4/5) but weak in Offers (2/5). Focus on anchoring next." Personalized curriculum: "Based on your EMPATHETIC Silhouette, prioritize firm language moves." Progression tracking across sessions.

Existing Data

5 confidence fields in user schema: confidencePreparation, confidenceDiscovery, confidenceOffers, confidencePushback, confidenceClosing. All 1-5 scale. Snapshot only — no historical tracking. No audit trail of how confidences change over time.

Stakeholder Input Needed

Are these 5 confidence dimensions (Preparation, Discovery, Offers, Pushback, Closing) the right axes? What other negotiation competencies should Ali track? How does Aligned measure growth with enterprise clients today — rubric, survey, observation?

Component 06

Session Replay

Tier 3
Methodology Root

Reflection-based learning — Ali coaching methodology emphasizes "discover your own insights." Session replay lets users review: what did Ali say? Why? How did my thinking change? Tied to all 4 phases. Also enables Tom (expert) to review user sessions and provide human coaching on top of Ali.

What Ali Generates

Annotated transcript with fold-out reasoning: which signal did Ali extract? Which Aligned Move was applied? Summary: "TRADING conversation, COMMUNICATION phase. Moves applied: The Power Map, Trade Cheap for Expensive." Three inflection types: Strong Move, Missed Opportunity, Coaching Moment.

Existing Data

Full conversation stored in ChatMessage with timestamps. metadata JSON captures coaching reasoning (phase, classification, entities). MessageFeedback tracks helpful/not-helpful ratings. Move invocations not tracked — moves are implicit in Ali's reasoning, not annotated.

Stakeholder Input Needed

When Tom reviews a coaching transcript, what does he look for? What patterns indicate "good coaching" vs. "missed opportunity"? Would Tom use this to review Chelsea's AI-assisted sessions? This determines whether Session Replay is a user feature or an admin/expert tool.
