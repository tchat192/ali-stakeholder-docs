1. Why — Problem Statement

The User Problem

Tom described it in the 03-02 sync: "I know very little about the beef commodity industry. There's terminology in the contract I had no context of what it means."

This is the problem every negotiation practitioner faces when a client hands them:

  • A legal contract full of domain-specific language
  • A set of commercial terms they need to negotiate
  • Zero preparation structure

Today, Tom's workflow is manual:

  1. Client sends raw contracts + context
  2. Tom reads everything (hours of work)
  3. Tom fills out his 4-phase planning workbook (the BBX Planning Tool) by hand
  4. Tom builds a negotiation strategy from the workbook

The contract decomposition module automates steps 1-3. It reads the contract, explains the domain terminology, and pre-populates Tom's planning framework — so the human expert spends time on strategy, not data extraction.

Why Now

  • Tom sent 4 real client files (anonymized and ready for development)
  • Chelsea (new Aligned employee) is the perfect dogfooding candidate — domain novice, needs the tool to consult effectively
  • Beta launch is imminent — a "coming soon" capability shows platform depth
  • Tom's 03-02 recommendation: "We can iterate in the background with just us... then flip the switch when it's commercializable"

Business Objective

Reduce contract-to-strategy preparation time from 4-8 hours (manual) to 30 minutes (AI-assisted review). Internal dogfooding validates before any external release.

2. Target Users

User Context Pain Point
Tom Hilgeman (SME) Receives client contracts across unfamiliar industries Must manually read, extract terms, fill planning workbook. Hours of prep before real advisory work begins.

Primary user for dogfooding: Chelsea. If a domain novice can use the output to prepare for a client engagement, the module works.

3. Feature Capabilities

What the Module Does

The module takes raw contract documents and produces a pre-populated negotiation planning workbook — structured to match Tom's existing 4-phase methodology.

Input

One or more documents:

  • Legal contracts (PDF, DOCX)
  • Commercial agreements / variations
  • Planning spreadsheets (XLSX)
  • Counterparty correspondence (DOCX, TXT)

Output: Negotiation Planning Brief

The output maps directly to Tom's BBX Planning Tool structure (his existing methodology):

US-2: Pre-Populated Planning Workbook

As an Aligned consultant, I want the contract data pre-populated into Tom's 4-phase planning framework, So that I can start strategic work instead of data entry.

Acceptance Criteria:

  • [ ] Output includes all 4 phases (Prepare, Communicate, Propose, Align)
  • [ ] Terms list is automatically extracted with Get/Give/Both classification
  • [ ] Priority scoring (HIGH/MED/LOW) matches Tom's methodology (clause emphasis + penalty structure)
  • [ ] BATNA sections are PRESENT in the schema with source: "awaiting_input" — transparent about needing manual input, not left blank or hidden
  • [ ] Sections with insufficient data show what Ali found, what Ali DIDN'T find, and what the user should fill manually (Pattern Mirror)
  • [ ] Tom validates output accuracy on 3+ real contracts at >70% field accuracy

US-3: Domain Glossary

As a consultant unfamiliar with the client's industry, I want industry-specific terms explained in plain language with negotiation context, So that I understand what I'm negotiating without external research.

Acceptance Criteria:

  • [ ] Identifies 80%+ of domain-specific terms (verified against Tom's manual annotations)
  • [ ] Each term has: definition, negotiation significance, related contract section
  • [ ] No hallucinated definitions — sources cited or marked "[confidence: low]"
  • [ ] Glossary is generated alongside the planning workbook, not as separate step

US-4: Approach Classification

As a consultant, I want Ali to classify the negotiation type, So that I know which of the 35 Aligned Moves are relevant to this deal.

Acceptance Criteria:

  • [ ] Classifies into one of 5 types: WEAK_POSITION, BARGAINING, TRADING, CREATING, PARTNERING
  • [ ] Provides confidence score (0-1) with reasoning
  • [ ] Surfaces top 5-8 relevant Aligned Moves with explanation of why each applies
  • [ ] Tom validates classification accuracy on 5+ real contracts

5. Failure Modes

Scenario Impact System Behavior Mitigation
Scanned PDF with poor OCR quality Garbled text → wrong extractions Show OCR confidence score. If <70% per-page, flag as "low confidence" and highlight which pages. Allow user to paste corrected text. Already have Tesseract OCR at 300dpi. Could add per-page confidence scoring.
Contract in unfamiliar legal jurisdiction Terms may have different meanings (US vs. AU vs. UK law) Detect jurisdiction from governing law clause. Flag jurisdiction-specific terms. Default to US if unknown. Governing law extraction is in regex pipeline.
Multi-party contract (not bilateral) Planning framework assumes 2 parties Identify all parties. If >2, ask user to specify "our side" and "their side." Map remaining parties. Party detection handles this via preamble parsing.
Contract with no financial terms Can't populate walkaway/goal zones Mark financial sections "[No financial terms found — manual input required]." Don't hallucinate numbers. Regex already handles this (count $-amounts found).
Confidential data leaks to API NDA violation, trust destruction Anonymize ALL files locally first. Never send raw documents to any API. Mapping file stays local only. Anonymize script (Stage 1) is the hard gate. No bypass.
LLM hallucinates contract terms User prepares for wrong negotiation Every extracted field links back to source section/page. No field exists without a citation. User can click-through to verify. Source grounding is a hard requirement, not nice-to-have.
Domain glossary is wrong Consultant misunderstands industry terms, loses credibility Mark confidence level on each definition. For low-confidence terms, say "This term may mean [X] in [industry]. Verify with client." Separate glossary confidence from extraction confidence.
Chelsea (novice) trusts output without verification Acts on incorrect analysis Include a "Review Required" checklist at top of output. Critical fields (walkaway, BATNA) are always flagged for human review. UI pattern: red banner "This is AI-generated. Verify before use."
Tom's methodology changes Output framework is stale Planning framework structure is configurable, not hardcoded. Phases/sections come from a config file Tom can update. negotiation-framework.json config that maps extraction → output sections.

6. Technical Architecture

Architecture rationale: A dialectic analysis (Round 1: domain-native vs. platform transplant; Round 2: contract-only vs. multi-source) produced two synthesis insights that reshape this architecture:

  1. Schema-Driven Extraction — Tom's methodology IS the extraction schema, not a downstream consumer. This collapses the original 5-stage pipeline to 3 stages.
  2. Anticipatory Schema — Design for multi-source intelligence from day one, implement contract-only in MVP. "Design for the constellation, implement the first star."

Pipeline (3 Stages)

Raw Files → [1] Anonymize → [2] Schema-Driven Extract → [3] Coach
              (EXISTS)        (BUILD — the product)       (BUILD)
                │                      │                      │
          Local PII strip      Commodity parsing +      Ali's moat
          (already built)      Negotiation schema       (Pattern Mirror,
                               (Docling/LlamaParse      coaching engine,
                                parses; LLM extracts    simulation)
                                to Tom's schema)

Why 3 stages instead of 5: The original pipeline (Anonymize → Extract → Classify → Analyze → Output) treated extraction, classification, and analysis as separate sequential steps. But Tom's methodology doesn't separate these — when he reads a contract, he simultaneously identifies terms, classifies their value flow, and assesses leverage. The negotiation extraction schema does this in a single LLM pass driven by the methodology itself.

Stage 1: Anonymize (Done — scripts/anonymize-contract.py)

  • Local-only PII stripping, OCR for scanned PDFs (Tesseract 300dpi)
  • --parties flag for explicit name redaction
  • Outputs: anonymized text + JSON mapping
  • Hard gate: no raw documents ever leave the user's machine

Stage 2: Schema-Driven Extract (Build — this IS the product)

  • Parsing is commodity. Use Docling (MIT, Linux Foundation) or LlamaParse (1K free pages/day) to convert PDFs/DOCX into structured text with section boundaries, tables, and reading order. This is domain-agnostic infrastructure — borrow freely.
  • Extraction is schema-driven. Feed the anonymized, parsed text to an LLM with Tom's negotiation methodology expressed as a structured extraction schema (see Section 6.1). The schema tells the LLM what to look for — negotiable terms, not legal clauses. The LLM performs term identification, Get/Give/Both classification, priority scoring, domain glossary generation, and approach classification in a single pass.
  • Per-field confidence + source citation. Every extracted field carries a confidence score (0.0-1.0) and a source reference (section/page in the contract). Inspired by Reducto's per-field confidence pattern.
  • Input: anonymized text. Output: structured JSON matching the Negotiation Extraction Schema (Section 6.1).

Stage 3: Coach (Build)

  • Format extraction output into Tom's 4-phase planning workbook structure
  • Apply Pattern Mirror: show what Ali sees, show what Ali DOESN'T see, let user fill gaps
  • Coaching assertiveness calibrated by source confidence (see Section 6.2)
  • Render as: Markdown (Notion), JSON (API), HTML (artifact)
  • Score relevance of 35 Aligned Moves against deal structure
  • Include source citations for every field

6.1 Negotiation Extraction Schema

This is the product primitive — Tom's methodology expressed as a machine-executable extraction prompt. Anyone can parse a PDF. Nobody has a schema that transforms clauses into negotiable terms with Get/Give/Both classification, priority scoring, walkaway zones, and trading potential.

{
  "negotiable_terms": [
    {
      "term_name": "string — the negotiable unit, may span multiple clauses",
      "related_clauses": ["clause references this term derives from"],
      "get_give_both": "GET | GIVE | BOTH",
      "current_position": "what the contract currently states",
      "flexibility_signals": ["language suggesting room for movement"],
      "leverage_indicators": ["factors that strengthen/weaken this position"],
      "priority_us": "HIGH | MEDIUM | LOW — with reasoning",
      "priority_them": "HIGH | MEDIUM | LOW — inferred from clause emphasis",
      "walkaway_zone": "if determinable from contract language",
      "trading_potential": "what this term could be packaged with",
      "confidence": 0.82,
      "source_reference": "Section 4.1-4.3, pages 12-14",
      "sources": [
        {
          "type": "contract",
          "reference": "Sections 4.1-4.3",
          "confidence": 0.82
        }
      ],
      "source_count": 1,
      "corroboration_status": "single_source",
      "coaching_assertiveness": "moderate"
    }
  ],
  "domain_glossary": [
    {
      "term": "industry-specific term found in contract",
      "definition": "plain-language meaning",
      "negotiation_significance": "why this matters for leverage/strategy",
      "source_section": "where in the document",
      "confidence": 0.9,
      "definition_source": "public_knowledge | inferred | low_confidence"
    }
  ],
  "approach_classification": {
    "type": "WEAK_POSITION | BARGAINING | TRADING | CREATING | PARTNERING",
    "confidence": 0.75,
    "evidence": ["signals from contract structure"],
    "sources": [{ "type": "contract", "reference": "overall structure" }],
    "source_count": 1
  },
  "behavioral_signals": {
    "collaborative_indicators": ["language suggesting partnership"],
    "adversarial_indicators": ["language suggesting zero-sum"],
    "power_asymmetry": "which party has structural advantage and why",
    "source": "contract_language_only",
    "coaching_note": "Behavioral assessment from contract language only. Email analysis (Phase 4) would strengthen this."
  },
  "batna": {
    "our_alternatives": [],
    "their_alternatives": [],
    "source": "awaiting_input",
    "confidence": null,
    "coaching_note": "BATNA requires market context. Enter alternatives manually or connect market data (Stage 5)."
  }
}

Key design decisions:

  • source_type on every field. Even in Sprint 1 when the only source is "contract," every field carries its provenance. When email intelligence arrives in a future phase, it slots into the same schema with a different source_type. This is cheap now (~1 string field) and architecturally essential.
  • source_count and corroboration_status. A classification from contract-only analysis has source_count: 1. When the same term is corroborated by an email or user input, source_count increases and confidence goes up. The coaching engine is designed to weight multi-corroborated insights higher.
  • Fields that contracts cannot fill are EXPLICIT, not absent. BATNA is in the schema. It starts as "source": "awaiting_input". Tom or Chelsea fills it manually. When market data integration arrives, it becomes "source": "market_api". The field exists from day one. The automation arrives later.
  • Terms ≠ Clauses. A "Compensation Package" term might span clauses 4.1 (Base Salary), 4.2 (Bonus), 4.3 (Equity), and 8.1 (Non-Compete). The related_clauses field captures this many-to-one mapping. This is the structural insight legal AI tools miss — they decompose into clauses, we decompose into negotiable terms.

6.2 Coaching Assertiveness Gradient

Source diversity determines how assertively Ali coaches:

Source Count Corroboration Status Coaching Assertiveness Example Coaching Language
1 (contract only) single_source Moderate "Based on the contract language, this term appears to be a Get."
2 (contract + user input) partially_corroborated Moderate-High "Based on the contract and your input about priorities, this is likely your strongest term."
3+ (contract + email + market) fully_corroborated High "Based on the contract, three prior emails, and market benchmarks, this is clearly your strongest leverage point."

In Sprint 1, everything is source_count: 1 and coaching is moderate. This is by design — not a limitation, but an honest assessment of confidence given available data.

11. Open Questions for Tom (Updated)

Original Questions (from v1)

  1. Walkaway/Goal zone methodology: In the planning tool, terms have Walkaway → Pessimistic → Goal → Optimistic zones. How do you determine these? Is there a formula, or is it expert judgment? (This determines whether Ali can auto-generate them or must leave them blank.)

  2. Get/Give/Both classification: What signals in a contract clause indicate whether a term is a "Get" vs. "Give" vs. "Both"? Is it purely based on which party it benefits, or is there nuance?

  3. Approach classification accuracy: The BBX deal was classified as "Concession Trading." What in the contract structure told you that? What would make it "Integrative" instead?

  4. Problem Planner prioritization: The tool has Impact x Likelihood scoring. Do you have a rubric for this, or is it intuitive? Can Ali pre-populate likely problems from contract risk clauses?

  5. Did you send the 3-4 bullet points on desired output that Terrance asked for in the 03-02 call? If not, this PRD is asking the same question in more detail.

New Questions (from dialectic analysis)

  1. Schema validation: We've expressed your methodology as a JSON extraction schema (Section 6.1). Does it capture how you actually decompose contracts? What's missing? What's wrong? (This is the #1 falsification condition — if the schema doesn't match Tom's methodology, the architecture is wrong.)

  2. Terms vs. clauses: We designed the schema so that a "Compensation Package" term spans clauses 4.1 (Base Salary), 4.2 (Bonus), 4.3 (Equity). Is this how you think about it? Or do you decompose differently?

  3. Empty fields for non-contract data: The schema includes BATNA, behavioral signals, and relationship history sections that start empty (marked "awaiting input") because contracts alone can't fill them. Would you and Chelsea actually fill these manually? Or are they clutter until we automate those inputs?

  4. Coaching confidence levels: When Ali coaches from contract-only data, it would say "Based on the contract language, this term appears to be a Get" (moderate confidence). When it has multiple sources, it would be more assertive. Does this make sense? Or would the moderate language undermine trust? (If users are confused by confidence levels, the whole assertiveness gradient is wrong.)

  5. Source expansion priorities: After contracts, we planned: term sheets → user-entered context → email analysis → market data. Does this sequence match how you'd prioritize expanding Ali's intelligence? What would you reorder?

Appendix: Tom's Planning Tool Structure (Sheets)

Extracted from the anonymized BBX Planning Tool:

  1. Dashboard — Negotiation overview, approach type, terms tracker, team rosters, favorability score

  8. 04 - Align (Record Keeping) — Position tracker per term over multiple rounds
  9. Notes — Current, internal, and historical negotiation notes
  10. Get/Give/Both Check — Scoring tool for term classification
  11. Problem Planner — Risk assessment matrix (up to 20 problems, impact x likelihood, mitigation)
