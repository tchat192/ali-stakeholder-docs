Overview
Why Profile Data Matters
Profile data solves Ali's cold-start problem. Without it, Ali has no context until a user shares details in conversation — which takes 2–3 exchanges. With profile data, Ali delivers personalized coaching from the very first message. S1
Principle 01

Cold Start Accelerator

Profile gives Ali context before conversation begins. A VP in healthcare gets different coaching than an entry-level engineer — instantly, on message one. S2

Principle 02

Conversation Wins

When conversation contradicts profile, Ali trusts the conversation. Profile is the starting calibration; real-time signals always override. S1

Principle 03

Progressive & Optional

Only 8 fields are required. Everything else is collected over time through Settings or in-app nudges. Users are never gated from coaching. S4

Data Handling
Privacy & Security Model
Profile data has three privacy tiers that control how it flows through the system.
Tier 1

Shared Professional Context

Always in system prompt. Name, role, industry, company, counterparts. Professional context for personalization. No PII beyond first name.

Tier 2

Calibrate Sensitive Business

In prompt but Ali never echoes values in responses. Deal values, constraints, approval limits. Used silently to calibrate, never to reference. S7

Tier 3

User Ctrl Identity & Personal

Only shared if user explicitly enables via toggle. Pronouns, communication needs, personal boundaries. Default: OFF.

What Ali Does Not Do

Does not share profile data with counterparts. Does not use data to train models for other users. Does not store deal specifics outside the account. Data encrypted at rest (Supabase). Anthropic does not train on API data.

For Stakeholder Review
Key Decisions
Please review and provide feedback on each decision below.

1. Conversation Always Wins

APPROVE / REJECT / MODIFY

When profile data contradicts what the user says in conversation, the conversation signal wins. Profile = starting calibration; real-time context is more accurate. S1

Example: Profile says "Healthcare" but user describes a software licensing deal — Ali coaches for software, not healthcare.

2. Eight Required Fields for MVP

APPROVE / REJECT / MODIFY
  1. role_title — Current role
  2. seniority_level — Seniority
  3. industry — Industry / sector
  4. company_size — Company size
  5. counterpart_types — Who you negotiate with
  6. archetype_name — At least 1 deal archetype
  7. current_focus — What you're currently negotiating
  8. primary_goals — Your top goals with Ali

Question: Is this the right set? Should any be optional? Should any optional fields become required?

3. Privacy Tiers

APPROVE / REJECT / MODIFY

Shared: always in prompt. Calibrate: in prompt, never echoed. User Controlled: opt-in only. See Privacy tab for field assignments.

Question: Are the tier assignments correct? Should any "Shared" fields be elevated to "Calibrate"?

4. Phase Sections 5–7?

APPROVE / REJECT / MODIFY

Sections 5–7 (Constraints, Learning Preferences, Accessibility) are enhancement-tier. Recommendation: collect through conversation and advanced Settings, not dedicated onboarding UI. Industry benchmarks: <10% completion for optional advanced sections.

Question: Agree with phased approach, or build all 7 sections upfront?

5. Deal Archetype Granularity

APPROVE / REJECT / MODIFY

Each deal archetype has 8 sub-fields (name, category, value range, duration, commercial levers, non-commercial levers, blockers, leverage). 1–5 archetypes per user.

Question: Right level of detail? Too much for users? Too little for Ali?
