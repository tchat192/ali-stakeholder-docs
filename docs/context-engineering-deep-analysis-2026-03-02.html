Executive Brief

  • The big idea: CSIRO Data61's "Everything is Context" paper proposes a Unix filesystem abstraction for agent memory — treating memory, tools, knowledge, and human input as mountable, governed context. This is the first serious attempt to systematize what we've been building ad-hoc in .claude/.
  • The critique that matters: John Maki (ex-Bain Capital, Serotonix CEO) argues the paper delivers "half the answer" — the infrastructure is right but it lacks a formal specification language for what goes IN the files. "Objects with states, processes transforming them, typed dependencies — that's conceptual modeling without a modeling language."
  • Production reality check: Synthetic benchmarks show 84-89% coding accuracy; production drops to 25-34%. ActionEngine proves 2-agent architectures beat multi-agent sprawl (95% success, 11.8x cheaper). Manus rebuilt their agent framework 4 times.
  • Our position: Ali's .claude/ architecture already implements 6 of 8 AFS concepts. The gaps (formal context schemas, cost-based selection, Context Evaluator, ACLs) are the engineering frontier. This document maps 12 concrete opportunities.
Paper Analysis

"Everything is Context" — CSIRO Data61

arXiv 2512.05470 • Xu, Mao, Bai, Gu, Li, Zhu • Dec 5, 2025

The paper extends the Unix principle "everything is a file" to AI systems: "everything is context." Instead of fragmented prompt engineering, RAG, and tool integration, all context artifacts are unified under a single file-system abstraction with uniform mounting, metadata governance, and access control.

The Formal AFS Specification

AFS = (N, E, M, meta, ACL) N = Nodes (files and directories) E = Edge relations (containment) M = Mount point mappings to external resources meta = Key-value metadata per node (ownership, timestamps, provenance, confidence, version) ACL = Access control lists (read / write / exec / admin)

Three Design Constraints

  1. Token Window (C1) — Hard architectural ceiling. Quadratic attention complexity (n²). Context must be curated, compressed, and incrementally streamed.
  2. Statelessness (C2) — LLMs retain nothing between calls. External persistent context repository required.
  3. Non-Determinism (C3) — Identical prompts yield varying outputs. Input-output pairs + metadata must be preserved for audit and replay.

Implementation: AIGNE Framework

Open-source TypeScript framework (337 stars, 183 forks). SQLite-backed. Multi-model support (OpenAI, Anthropic, Gemini, DeepSeek, Ollama). Native MCP integration. Elastic 2.0 license.

Key architecture choice: programmable resolvers with declarative mappings (GraphQL/OpenAPI-style schemas) that project any mounted resource into the file system namespace. Search integrates ripgrep (content) + Faiss (vector similarity) with sub-100ms latency.

Key Limitation

The paper is an architectural proposal, not an empirical evaluation. No quantitative benchmarks, no comparison baselines, no user studies. The AIGNE framework is real but early (337 stars). The value is in the conceptual model, not proven production results.

Critical Analysis

The "Objects with States" Critique

John W. Maki — CEO Serotonix, former Managing Director Audax Group, former Principal Bain Capital

"This paper is solving the right problem with half the answer. The infrastructure is spot-on. Context should be mounted, governed, and selectively loaded, not dumped into the window as a blob. But look at their memory taxonomy closely. They're describing objects with states, processes transforming them, and typed dependencies between stages. That's conceptual modeling without a modeling language. The gap between 'everything is a file' and 'everything is context' is a specification language for what's in those files. One already exists." — @John_W_Maki, March 2, 2026

Maki's Four-Layer Argument

EndorsementInfrastructure is correct — mount, govern, selectively loadThe AFS architecture itself is sound

Why This Matters for Ali

Our .claude/ architecture mirrors AFS: we have mounted context (SACRED.md, instructions.md), selective loading (startup.sh), memory hierarchy (Mem0 + Knowledge Graph). But we also lack formal schemas. Our spec.md is the closest thing to a "specification language" — it defines typed tasks with states (pending/complete), validation rules, and dependency tracking. Maki's critique suggests the spec reconciliation pattern is more architecturally important than we realized.

The Implied Specification Languages

Maki hints at "one [that] already exists." Candidates based on his background (PE/VC, pharma, AI governance):

  • OWL/RDF — Ontology Web Language. Formally describes classes, properties, constraints. Used in biomedical ontologies (his domain).
  • JSON Schema / JSON-LD — Lightweight typed schemas with linked data semantics. Most pragmatic for LLM tooling.
  • UML State Machines — Formal notation for objects with states, transitions, guards. Maps directly to memory lifecycle.
  • Alloy / TLA+ — Formal specification languages for verifying system properties.

For Ali MVP, JSON Schema + state machine notation is the pragmatic choice — it's already the lingua franca of LLM tool definitions (inputSchema, outputSchema).

Agent Memory

Memory Paradigm Shift

Three converging research threads redefine what "agent memory" means:

1. Execution Trajectories vs. Conversation Logs (DAIR.AI)

Traditional approach: save conversation history, retrieve relevant chunks. Emerging approach: save execution trajectories — what the agent did, which tools it called, what worked vs. failed, and the outcome. This is experiential memory (Type 4 in AFS taxonomy).

Ali Opportunity

Our WORKFLOW_PLAYBOOK.md is hand-curated experiential memory (P1-P37 patterns, A1-A29 anti-patterns). Automating this — extracting successful patterns from session transcripts — would be a significant uplevel.

2. Reflector-Driven Consolidation (Snapchat/Auton Agent)

Instead of raw memory accumulation, use a periodic reflector agent that consolidates, deduplicates, and prioritizes memories. The reflector runs between tasks (not during) and produces distilled knowledge.

This maps to our session-end protocol (reconcile spec, save to Mem0, update playbook). But the reflector concept suggests this should be automated and continuous, not just at session boundaries.

3. Context+ (Tree-Sitter AST + Spectral Clustering)

MCP server that solves context bloat by parsing code into AST nodes, then using spectral clustering to select only relevant code fragments. Instead of dumping whole files, it provides structurally relevant code slices.

Watch

Interesting for Ali's codebase but our current context is more documentation/specification-heavy than code-heavy. More relevant when we scale to multi-service architecture.

4. MemOS (May/Jul 2025)

Three memory types: Parametric (model weights), Activation (KV-cache), Plaintext (external). The "MemCube" abstraction enables cross-type scheduling and lifecycle management. V2 adds skill memory for cross-task reuse.

MemOS is more ambitious than AFS (it includes model weights as "memory"), but less practical for our use case. AFS's filesystem metaphor is more actionable for application-layer engineering.

Orchestration

Orchestration Layer Primitives

Annelies Gamble's framework identifies four primitives that every production AI system needs:

Intelligent Routing Directing queries to the right model/agent/tool Partial Skills system routes, but no dynamic model selection

Critical Gap

Telemetry is our biggest orchestration gap. We monitor uptime but not: which tools agents use, how much context they consume, which memories they access, or how often they hallucinate. AFS's "every interaction logged as a transaction" is the right model — we need structured agent telemetry before production beta.

ComposioHQ Agent Orchestrator Pattern

Parallel coding agents with git worktree isolation. Each agent works on a separate task in its own worktree, preventing merge conflicts. A coordinator reviews and merges. We already use worktrees via the using-git-worktrees skill — the insight is using them for parallel agent execution, not just developer isolation.

AI Web Browsing

ActionEngine: State Machine Memory Maps

Source: Microsoft + Georgia Tech research

ActionEngine demonstrates that 2-agent architectures beat multi-agent sprawl for web browsing tasks:

MetricActionEngine (2-agent)Multi-agent baselines
Success Rate95%60-80%
Cost11.8x cheaperBaseline
Training RequiredNone (training-free)Often required
ArchitectureState machine + memory mapsMulti-agent coordination

The Two-Agent Architecture

  1. Planner Agent — Decomposes tasks into state machine graphs with decision points
  2. Executor Agent — Follows the state machine, recording observations as "memory maps" for reuse

The state machine memory maps are reusable across similar tasks — once the system learns how to book a flight on United, it can replay that map with different parameters. This is procedural memory (Type 5 in AFS taxonomy) made concrete.

Ali Opportunity

Ali's negotiation coaching could use this pattern: build state machine maps of common negotiation patterns (salary negotiation, vendor contract, etc.) that agents traverse at runtime. The "Aligned Moves" in our codebase (35 documented) are essentially unstructured procedural memory — formalizing them as state machines would enable more reliable coaching.

Reality Check

The Benchmark-Production Gap

Source: arXiv 2510.26130

EnvironmentCoding AccuracyNotes
Synthetic Benchmarks84-89%Clean, isolated problems

Gap~55 percentage points3x worse in production
Implication

This validates Ali's "Ask Don't Infer" principle. When LLMs are 3x worse in production vs. benchmarks, the ethical choice is to surface uncertainty rather than project false confidence. Our anti-hallucination protocol (grounding markers, source attribution) is architecturally aligned with this reality.

AGENTS.md Anti-Pattern (arXiv 2602.11988)

Auto-generated project context files (like AGENTS.md) hurt performance by 3% and increase cost by 20%. The problem: generic context bloat. The lesson: curated context beats generated context. Our manually-maintained SACRED.md (200 tokens) and startup.sh (dynamically generated, focused) are the right approach.

Claude Code "Seeing Like an Agent"

Less is more. Simple composable tools beat specialized toolkits. This validates our skill system architecture: small, focused skills (/sprint, /deploy, /reconcile) over monolithic frameworks.

Production Lessons

Manus: 4 Framework Rebuilds

Manus (production AI agent platform) shared hard-won lessons from rebuilding their agent framework four times ("Stochastic Graduate Descent"):

Six Production Patterns

KV-Cache Optimization

100:1 input:output token ratio. Cached input is 10x cheaper than uncached. Design prompts for cache reuse.

Applicable Now

"Mask, Don't Remove"

Use logit masking to hide unavailable tools instead of removing them from prompts. Preserves cache, reduces cost.

Requires Tool Framework

Filesystem as Extended Context

Treat the filesystem as unlimited, persistent, directly operable context. Write intermediate results to files, not conversation.

Already Doing (artifacts, handoffs)

Attention Manipulation (todo.md)

Force recitation of task lists to keep goals in attention. Validated by our spec.md approach.

Already Doing

Error Retention as Learning

Keep error messages in context as signals. Don't just retry — let the model learn from failures within the session.

Already Doing (playbook anti-patterns)

Controlled Variation

Prevent pattern collapse by introducing controlled randomness. Avoid the agent settling into repetitive loops.

Not Implemented

What We're Missing (Per Maki's Critique)

  1. No formal specification language for context objects — SACRED.md and instructions.md are natural language, not typed schemas
  2. No typed dependencies between context stages — we don't formally define that episodic memory feeds fact memory feeds scratchpad
  3. No cost-based selection — we estimate token costs ("~5K context") but don't calculate per-fragment costs before loading
  4. No Context Evaluator — our grounding protocol is advisory, not enforcement. No automated hallucination detection.
  5. No ACLs on context files — a buggy agent could overwrite SACRED.md
Engineering Roadmap

12 Engineering Opportunities

Tier 1: High Impact, Low Effort (Do Now)

1. Context Object Schema

Define JSON Schema for each context type in .claude/: what fields exist, what's required, what states are valid. Start with spec.md task objects — they already have typed states.

Effort: 2-3h • Impact: High • Foundation

2. Token Budget Manifest

Modify startup.sh to output a JSON manifest: which context was loaded, estimated token cost, what was excluded and why. Enables cost-based optimization.

Effort: 1-2h • Impact: Medium • Observability

3. Automated Playbook Extraction

Build a session-end script that scans the conversation transcript for tool call patterns, extracts success/failure sequences, and proposes new playbook entries.

Effort: 3-4h • Impact: High • Memory

4. Memory Decay Dashboard

Query Mem0 for memories by age, flag those approaching expiration, surface which memories are referenced frequently vs. never.

Effort: 2h • Impact: Medium • Observability

Tier 2: Medium Effort, Strategic Value (Next Sprint)

5. State Machine Negotiation Maps

Formalize the 35 Aligned Moves as state machine nodes with transitions, preconditions, and expected outcomes. Enables systematic coaching paths.

Effort: 6-8h • Impact: Very High • Product

6. Context Evaluator v1

Post-response validation: compare AI output against loaded context sources. Flag claims without source attribution. Log confidence scores.

Effort: 4-6h • Impact: High • Safety

7. Agent Telemetry Pipeline

Structured logging of: tools invoked, context consumed, memory accessed, hallucinations detected. Output to JSON logs for analysis.

Effort: 4-5h • Impact: High • Observability

8. Reflector Agent (Periodic Memory Consolidation)

Scheduled agent that reviews recent Mem0 entries, deduplicates, resolves contradictions, and promotes important memories. Runs daily or on-demand.

Effort: 5-6h • Impact: High • Memory

Tier 3: Higher Effort, Architectural (Post-Beta)

9. Formal Context Mounting Protocol

Implement AFS-style mount points: each context source declares its schema, token cost, and access policy. The context constructor selects within budget.

Effort: 8-10h • Impact: Very High • Architecture

10. Multi-Agent Context Isolation

When spawning subagents, define sandboxed context scopes. Prevent research agents from accessing deployment credentials. Prevent tool agents from modifying governance files.

Effort: 6-8h • Impact: High • Security

11. KV-Cache-Aware Prompt Design

Restructure system prompts so the static portion (SACRED.md, persona) comes first and stays constant. Dynamic content (startup.sh output) is appended. This maximizes cache hits.

Effort: 3-4h • Impact: Medium (cost) • Optimization

12. Context Specification Language

Maki's "missing half." Define a JSON Schema-based DSL for context objects: typed fields, valid state transitions, transformation rules, dependency graph. Would make .claude/ programmatically verifiable.

Effort: 10-15h • Impact: Very High • Architecture

Priority Recommendation

Start Here

Opportunities 1 + 3 + 7 form a natural cluster: define context schemas, automate pattern extraction, and add telemetry. Together they create the observability layer needed to make informed decisions about the Tier 2 and Tier 3 investments. Total effort: ~8-10h. These three changes would move us from "ad-hoc context management" to "instrumented context infrastructure."
